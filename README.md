# RobinHood P2P v3.2

Децентрализованный P2P-мессенджер с оконечным шифрованием, работающий в браузере. Без установки, без центрального сервера хранения сообщений.

## Архитектура

Приложение использует три уровня связи:

| Уровень | Технология | Назначение |
|---------|-----------|------------|
| Сигнальный | WebSocket (Cloudflare Worker) | Обмен маяками, SDP, ICE-кандидатами. Сервер не хранит сообщения |
| Транспортный | WebRTC DataChannel + DHT | Прямой туннель между пирами после рукопожатия |
| Резервный | WebSocket (fallback) | Если DHT-канал недоступен, сообщения идут через WS |

DHT (Kademlia-подобная, 256 корзин, XOR-метрика, k=20, α=3) используется для поиска пиров и передачи сообщений в обход сигнального сервера.
Пользователь А ──WS──► Cloudflare Worker ◄──WS── Пользователь Б
│ │
└──────── WebRTC DataChannel / DHT ──────────┘
(прямой туннель)

**Слепой Улей:** Cloudflare Worker не читает содержимое сообщений, не хранит историю, не знает кто с кем общается. Он только пересылает зашифрованные блобы между подписчиками. Rate limiting: 30 сообщений/сек/пир.

## Идентификация

**Peer ID** = первые 32 символа SHA-256 от конкатенации:

- Параметров WebGL-рендерера (вендор + модель GPU)
- Характеристик AudioContext (частота дискретизации, число каналов)
- Размеров экрана и глубины цвета
- `navigator.hardwareConcurrency` и `deviceMemory`
- Случайной соли (генерируется однократно, хранится в localStorage)

Это даёт стабильный, но не привязанный к личности идентификатор устройства.

## Сквозное шифрование (E2EE)

### Рукопожатие
ECDH на кривой P-256. Каждая сторона генерирует ephemeral-пару, обменивается публичными ключами через маяки (blind beacon).

### Double Ratchet + HMAC
После рукопожатия применяется Double Ratchet:
- **Ratchet Key** обновляется каждое сообщение: `SHA-256(старый_ключ + индекс)`
- **HMAC-SHA256** подписывает каждое сообщение перед шифрованием
- **Старые ключи** сохраняются (до 50) для расшифровки сообщений, пришедших не по порядку
- **Ресинхронизация:** при расхождении ratchet запрашивается автоматический перезапуск

### Шифрование сообщений
- **AES-256-GCM**, случайный IV (12 байт)
- **GZIP-сжатие** перед шифрованием
- **Случайный паддинг** (20–70 байт) внутри сообщения
- **Фиксированный размер блоба:** 4096 байт (padding против анализа размера)

### Индикатор Ratchet
В статус-баре отображается количество оборотов ratchet:
- 🔄 0–9 — инициализация
- 🔒 10–49 — активный
- 🔐 50+ — безопасный

## Маяки и каналы

| Сущность | Время жизни | Описание |
|----------|-------------|----------|
| Маяк | 5 минут | Зашифрованное объявление с targetPeerId |
| Канал | 10 минут (продлевается активностью) | Зашифрованный туннель между двумя пирами |
| Колчан | — | Метафора для UI подключения |

### Процесс подключения

1. **Устройство А:** открывает Колчан (кнопка 🏹), вводит Peer ID друга, жмёт «Создать маяк»
2. **Устройство Б:** получает маяк, автоматически создаёт канал, отправляет beacon-response
3. На устройстве Б загорается кнопка «Заявки» — пользователь нажимает для подтверждения
4. На устройстве А приходит beacon-response — канал создан
5. **Оба видят:** `✅ Канал открыт!`, контакт появляется в Лучниках

## Сообщения и звонки

### Текстовые сообщения
Шифруются блобом, отправляются через DHT (предпочтительно) или WS (fallback). При получении **автоматически отображаются** в чате — не требуется вручную выбирать контакт.

### Голосовые сообщения
- Запись через MediaRecorder (Opus в WebM, 16 kbps)
- Отображаются с аудиоплеером (кнопка play/pause, волновая анимация, таймер)
- Шифрование и отправка блобом с тегом `voice: true`

### Голосовые вызовы
- Полный WebRTC-стек с рингтоном (`melodi.mp3`) и КПВ (`Welk.mp3`)
- Кнопки «Ответить»/«Отклонить» при входящем вызове
- ICE-кандидаты буферизируются и отправляются группой
- Авто-установка активного канала при входящем звонке
