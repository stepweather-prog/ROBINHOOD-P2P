# RobinHood P2P v2.4.1

Децентрализованный P2P-мессенджер с оконечным шифрованием, работающий в браузере. Без установки, без центрального сервера хранения сообщений.

## Архитектура

Приложение использует два уровня связи:

| Уровень | Технология | Назначение |
|---------|-----------|------------|
| Сигнальный | WebSocket (wss://...) | Только для первого касания: обмен SDP, ICE-кандидатами, маяками |
| Транспортный | WebRTC DataChannel + DHT | Прямой туннель между пирами после рукопожатия |

DHT (Kademlia-подобная, 256 корзин, XOR-метрика, k=20, α=3) используется для поиска пиров и передачи сообщений в обход сигнального сервера. Если прямой DHT-канал недоступен, сообщение прозрачно уходит через WebSocket — без разрыва для пользователя.
Пользователь А ──WS──► Сигнальный сервер ◄──WS── Пользователь Б
│ │
└──────── WebRTC DataChannel / DHT ──────────┘

## Идентификация

**Peer ID** = первые 32 символа SHA-256 от конкатенации:

- Параметров WebGL-рендерера (вендор + модель GPU)
- Характеристик AudioContext (частота дискретизации, число каналов)
- Размеров экрана и глубины цвета
- `navigator.hardwareConcurrency` и `deviceMemory`
- Случайной соли (генерируется однократно, хранится в localStorage)

Это даёт стабильный, но не привязанный к личности идентификатор устройства.

## Сквозное шифрование (E2EE)

**Рукопожатие:** ECDH на кривой P-256. Каждая сторона генерирует ephemeral-пару, обменивается публичными ключами через сигнальный сервер.

**Общий секрет:** Вычисляется через `deriveBits()`, 256 бит.

**Шифрование сообщений:** AES-256-GCM, случайный IV (12 байт), IV предшествует шифротексту.

**Блобы:** JSON сообщения дополняется случайными байтами до 4096 байт (padding против анализа размера), затем шифруется.

## Проверка ключа (Safety Number)

Из SHA-256 хеша общего секрета канала вычисляется проверочный код:

- 4 группы по 4 hex-символа (например: `a3f2-8b1c-440e-d91a`)
- 4 эмодзи, выбранных из хеша

Этот код предназначен для внеполосной верификации (OOB): пользователи сверяют его по альтернативному каналу связи (звонок, личная встреча). При подтверждении ключ помечается как `✅ ECDH`.

**Критическая деталь:** если при переподключении секрет канала изменился (новое рукопожатие дало другой ключ), приложение явно предупреждает пользователя: `⚠ Ключ канала изменился!`. Без этого OOB-верификация теряет смысл.

## Маяки и каналы

| Сущность | Время жизни | Описание |
|----------|-------------|----------|
| Маяк | 30 минут | Публичное объявление о готовности к соединению |
| Канал | 10 минут (продлевается активностью) | Зашифрованный туннель между двумя пирами |
| Колчан | — | Метафора для хранения активных каналов |

Маяк создаётся через кнопку `🏹 Создать маяк`, после чего отправитель может передать свой Peer ID получателю любым способом. Получатель вставляет Peer ID в поле «Подключиться» — происходит ECDH-рукопожатие и создаётся канал.

## Сообщения и голос

- **Текст:** шифруется как блоб, отправляется через DHT (предпочтительно) или WS (fallback).
- **Голосовые сообщения:** запись через MediaRecorder (Opus в WebM), шифрование, отправка блобом с тегом `voice: true`.
- **Голосовые вызовы:** полный WebRTC-стек с рингтоном (`melodi.mp3`) и КПВ (`Welk.mp3`). Кнопки «Ответить»/«Отклонить» при входящем вызове.

Три визуальные темы, переключаемые в «Опциях»:

| Тема | CSS-класс | Характер |
|------|-----------|----------|
| Бумага | `[data-theme="paper"]` | Светлая, минималистичная (по умолчанию) |
| Лес | `[data-theme="forest"]` | Тёмная, золотой акцент |
| Утро | `[data-theme="morning"]` | Тёплая пастельная |

## Требования

- **HTTPS** (обязательно для Web Crypto API)
- Современный браузер с поддержкой WebRTC и MediaRecorder
