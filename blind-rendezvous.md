# Техники слепого рандеву для P2PPong

## Введение

Текущая архитектура P2PPong использует сигнальный сервер как точку рандеву: создатель зажигает маяк по ключу `waiting_<beaconId>`, джойнер приходит на тот же ключ и забирает маяк. Сервер не знает содержимого пакетов (благодаря оконечному шифрованию), но видит:

- IP-адреса создателя и джойнера
- Временные метки запросов
- Факт обращения двух сторон к одному ключу

Этого достаточно, чтобы установить факт связи между двумя устройствами. Задача слепого рандеву — лишить сервер даже этой метаинформации.

Ниже рассмотрены три техники полной слепоты сервера.

---

## 1. Общий пул маяков (Blind Rendezvous)

### Принцип

Сервер хранит все маяки в общем пуле. Создатель помещает маяк в пул. Джойнер запрашивает у сервера **все** активные маяки и пытается расшифровать каждый своим приватным ключом. Только один маяк расшифруется успешно — это искомый.

Сервер не знает:
- Какой маяк кому принадлежит (маяки не имеют идентификаторов, только порядковые номера)
- Какой маяк запросил конкретный джойнер (он скачивает все)
- Состоялся ли контакт (джойнер не сообщает серверу об успехе)

### Реализация

**Сервер:**
- Хранит маяки в массиве без ключей
- Добавление маяка: `POST /pool` с телом `{ packet }`, возвращает `{ id: порядковый_номер }`
- Получение всех маяков: `GET /pool` возвращает `{ beacons: [{ id, packet }] }`
- Удаление маяка: `DELETE /pool?id=...` (опционально, маяки самоудаляются по TTL)
- Максимальный размер пула: 100 маяков
- TTL маяка: 5 минут

**Создатель:**
1. Формирует маяк как обычно
2. Отправляет `POST /pool` с маяком
3. Передаёт джойнеру `beaconId` вне полосы (через QR/голос/текст)

**Джойнер:**
1. Получает `GET /pool` — все маяки
2. Для каждого маяка пытается расшифровать `inner` своим приватным ключом
3. Успешно расшифрованный маяк содержит `peerId` создателя, `code`, `beaconId`
4. Сравнивает `beaconId` с полученным вне полосы — подтверждение подлинности

### Плюсы

- **Полная слепота сервера**: сервер не знает ни кто создал маяк, ни кто его искал, ни состоялся ли контакт
- **Устойчивость к анализу трафика**: джойнер всегда скачивает все маяки, его поведение неотличимо от любого другого джойнера
- **Простота серверной части**: тупой массив, никакой логики маршрутизации

### Минусы

- **Нагрузка на джойнера**: O(n) расшифровок, где n — количество маяков в пуле. При 100 маяках и времени расшифровки ~10ms — около 1 секунды
- **Предел масштабирования**: при большом количестве пользователей пул переполняется. Нужна либо шардирование пулов, либо ограничение
- **Расшифровка раскрывает факт контакта**: джойнер узнаёт, кто создатель, только после успешной расшифровки. Но сервер этого не видит

### Оценка сложности реализации

- Сервер: 2 часа (замена key-value на массив с TTL)
- Клиент: 4 часа (цикл расшифровки, проверка beaconId)
- Общая: 6 часов

### Оценка влияния на latency

- Текущая: 1 запрос к серверу (~100ms)
- С пулом из 100 маяков: 1 запрос + 100 расшифровок (~1-2 секунды)
- Неприемлемо для мобильных устройств при 100 маяках, приемлемо при 20-30

## 2. Хэш от общего секрета как идентификатор

### Принцип

Вместо случайного `beaconId` использовать `SHA256(общий_секрет + соль)`. Общий секрет известен только создателю и джойнеру (например, пароль, которым они обменялись вне полосы). Сервер видит ключ, но не может связать его с конкретными пользователями без знания секрета.

### Реализация

**Создатель:**
1. Генерирует случайную соль `salt`
2. Вычисляет `beaconId = SHA256(общий_секрет + salt)`
3. Зажигает маяк с ключом `waiting_<beaconId>`
4. Передаёт джойнеру `salt` (открыто, можно через сервер) и `общий_секрет` (вне полосы)

**Джойнер:**
1. Получает `salt` (из маяка или отдельно)
2. Вычисляет `beaconId = SHA256(общий_секрет + salt)`
3. Запрашивает маяк по ключу `waiting_<beaconId>`

### Плюсы

- **Простота реализации**: минимальные изменения в текущей архитектуре
- **Слепота сервера**: сервер видит только хэш, не может связать его с пользователями
- **Низкая latency**: 1 запрос, как сейчас

### Минусы

- **Требуется предварительный обмен секретом**: создатель и джойнер должны договориться о пароле вне полосы (встреча, звонок, QR)
- **Секрет может быть скомпрометирован**: если секрет утекает, сервер может вычислить beaconId и связать пользователей
- **Не решает проблему корреляции по IP и времени**: сервер всё ещё видит, что два IP обратились к одному ключу в близкое время

### Оценка сложности реализации

- Сервер: 0 часов (без изменений)
- Клиент: 2 часа (хэширование, UI для ввода секрета)
- Общая: 2 часа

### Оценка влияния на latency

- Без изменений: ~100ms
- Добавляется одно хэширование SHA256: ~1ms

## 3. Bloom-фильтр + ложные запросы

### Принцип

Сервер возвращает маяки, отфильтрованные через Bloom-фильтр. Джойнер отправляет серверу Bloom-фильтр, содержащий несколько возможных `beaconId` (включая искомый). Сервер возвращает все маяки, чьи идентификаторы проходят через фильтр. Джойнер делает несколько запросов с разными случайными фильтрами, чтобы скрыть, какой маяк ищет на самом деле.

### Реализация

**Создатель:**
1. Генерирует `beaconId` как обычно
2. Зажигает маяк с ключом `waiting_<beaconId>`

**Джойнер:**
1. Формирует Bloom-фильтр, содержащий:
   - Искомый `beaconId`
   - N случайных фиктивных `beaconId` (где N = 5-10)
2. Отправляет фильтр серверу: `POST /search` с телом `{ filter: base64_bloom_filter }`
3. Сервер возвращает все маяки, чьи `beaconId` проходят через фильтр
4. Джойнер повторяет запрос 2-3 раза с разными наборами фиктивных ID
5. Искомый маяк будет присутствовать во всех ответах, фиктивные — только в некоторых
6. Джойнер находит пересечение и забирает маяк

**Сервер:**
- Хранит маяки с ключами как сейчас
- `POST /search`: принимает Bloom-фильтр, возвращает массив маяков, прошедших фильтр
- Не знает, какой из возвращённых маяков искомый

### Плюсы

- **Компромисс между нагрузкой и слепотой**: джойнер получает не все маяки, а только прошедшие фильтр (обычно 5-10)
- **Сервер не может определить искомый маяк**: фильтр содержит несколько ID, ответ содержит несколько маяков
- **Гибкость**: можно регулировать N (количество фиктивных ID) и количество запросов

### Минусы

- **Сложность реализации**: Bloom-фильтр на клиенте и сервере, логика пересечения множеств
- **Ложноположительные срабатывания**: Bloom-фильтр может вернуть маяк, который не был в фильтре (вероятность ~1% при правильной настройке)
- **Несколько запросов**: увеличивает latency (3 запроса вместо 1)
- **Сервер видит несколько запросов от одного IP**: может коррелировать по времени и IP

### Оценка сложности реализации

- Сервер: 4 часа (Bloom-фильтр, эндпоинт `/search`)
- Клиент: 6 часов (Bloom-фильтр, генерация фиктивных ID, логика пересечения)
- Общая: 10 часов

### Оценка влияния на latency

- Текущая: 1 запрос (~100ms)
- С Bloom-фильтром: 3 запроса (~300ms) + обработка (~50ms) = ~350ms
- Приемлемо для большинства сценариев

## Рекомендация

Для третьего цикла разработки рекомендуется внедрить **комбинацию Техники 1 (Общий пул) и Техники 2 (Хэш от секрета)**:

1. **Основной режим — Общий пул маяков**:
   - Сервер хранит до 100 маяков в общем пуле
   - Джойнер скачивает все маяки и расшифровывает
   - Подходит для публичных каналов, где не требуется предварительный обмен секретом

2. **Режим повышенной безопасности — Хэш от секрета**:
   - Если пользователи хотят гарантировать, что сервер не увидит даже факт обращения к одному ключу
   - Требует предварительного обмена паролем (QR, голос, встреча)
   - `beaconId = SHA256(пароль + соль)`
   - Сервер видит только хэш, не может связать с пользователями
   - Может работать поверх Общего пула или отдельно

3. **Bloom-фильтр оставить на будущее**:
   - Как оптимизация Общего пула при росте числа пользователей >100
   - Когда latency станет проблемой (пул >50 маяков)

### План внедрения (третий цикл)

1. **Сервер**: добавить эндпоинты `/pool` (POST, GET, DELETE)
2. **Клиент**: режим «Публичный колчан» — использование общего пула
3. **Клиент**: режим «Тайный колчан» — хэш от секрета с UI для ввода пароля
4. **Тестирование**: сравнение latency и надёжности для обоих режимов

## Заключение

Полная слепота сервера достижима. Текущая архитектура (beaconId как ключ) — компромисс между простотой и слепотой. Следующий шаг — общий пул маяков — устраняет возможность сервера связывать создателя и джойнера по ключу. Хэш от секрета добавляет защиту от корреляции по IP и времени. Обе техники могут сосуществовать как два режима работы приложения.
