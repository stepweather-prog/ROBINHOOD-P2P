# Пакет для независимого криптоаудита P2PPong v6.3 / HTTPR v1.0

## 📦 Объём аудита

Проверке подлежат исключительно криптографические и транспортные компоненты ядра:
1. `crypto-worker.js` (v3.0) — реализация Web Crypto API, ECDH P-256, AES-256-GCM, HMAC-SHA256, HKDF, логика Triple Ratchet (Symmetric + DH).
2. `p2ppong.js` (v6.2) — протокол рукопожатия, генерация одноразовых сессий, раздельные цепи Send/Recv, обработка скользящего окна ключей.
3. `httpr-core.js` (v1.0) & `httpr-p2ppong-bridge.js` (v1.3) — транспортный мост, конвертное шифрование, агрегация плагинов и логика слепой маршрутизации.
4. `worker.js` / `server.js` — бэкенд-реализация Blind Locker (слепой ячейки) и анонимного публичного пула `/pool` на предмет утечки метаданных.

## 🚫 Не подлежат проверке
* `robinhood-ui.js` (интерфейс, 12 тем оформления, Lottie-анимации, Web Speech API).
* `peer-help.js`, `sw.js`, `index.html`, `manifest.json`.

## ❓ Ключевые вопросы аудитору

### 1. Криптографические примитивы и Triple Ratchet
* **Энтропия:** Достаточна ли стойкость генерации `Peer ID` и `Beacon ID` (32 случайных hex-символа через `crypto.getRandomValues`) для предотвращения коллизий и предсказуемости?
* **Реализация Triple Ratchet:** Насколько корректно развернута Signal-совместимая схема? Обеспечивает ли Symmetric Ratchet (через HKDF/SHA-256) полноценную *Forward Secrecy* (немедленное уничтожение отправленных ключей) и гарантирует ли DH Ratchet (смена корня каждые 10 сообщений) заявленную *Post-Compromise Security*?
* **Кэширование старых ключей:** Безопасен ли кэш глубиной в 3 шага для отложенных `receiving-ключей`? Нет ли векторов для вечной утечки памяти (Memory Leak) или атак на истощение ресурсов Web Worker при отправке пакетов со специально искаженными индексами `ri`?

### 2. HTTPR Конверты и Транспортный мост
* **Деривация транспортного ключа:** Достаточно ли безопасен метод `AES-256-GCM` на транспортном ключе `SHA-256(secret + "transport")` при использовании режима «Тайного колчана»? Существует ли риск быстрой атаки по словарю (Brute-force на GPU) на текстовый секрет, если структура конверта (`v`, `tid`, `hop`, `ttl`, `ts`) полностью открыта для транспорта?
* **Целостность конверта:** Исключает ли валидация `AES-256-GCM` на слое полезной нагрузки (`pl`) возможность MitM-модификации транзитных полей пакета без нарушения криптографической целостности?
* **Атака на авто-ресинхронизацию:** Безопасна ли функция `advanceRecvRatchet`? Может ли злоумышленник, имеющий доступ к сигнальному серверу, симулировать расхождение индексов и принудительно заставить ядро бесконечно прокручивать цепь или сбросить состояние сессии?

### 3. Метаданные и Анонимность (Бэкенд)
* **Анализ трафика на `/pool`:** Какие объёмы метаданных (IP, паттерны таймингов запросов, частота обращений) остаются доступны Cloudflare Worker / Render Server в каждом из трёх режимов («Обычный маяк», «Тайный колчан», «Публичный колчан»)? Насколько эффективно Blind Pool (ограничение до 100 маяков, TTL 5 минут) нивелирует корреляцию пиров?
* **Атака на код верификации:** Насколько устойчив 7-значный цифровой код (`~23 бит` энтропии), передаваемый через сигнальный сервер во время рукопожатия, к атакам перебора на лету (Online Brute-force) в течение 5-минутного окна жизни маяка?
